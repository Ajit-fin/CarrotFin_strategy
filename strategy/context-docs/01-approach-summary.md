## What CarrotFin Is

CarrotFin is an AI-native personal finance advisor for Indian consumers. The AI doesn't assist — it **runs** the experience. Every screen, recommendation, and interaction is generated by the AI based on who the user is, what they need right now, and what they should do next. It integrates conversational AI with adaptive, generative UI into a single surface.

**Not** a robo-advisor, not a dashboard with a chatbot, not an incremental improvement over existing apps.

---

## Core Architectural Approach

### Three Design Axioms

1. **Adaptive Composition** — The UI is composed at render time by the AI from a palette of components and composition rules. The interface restructures itself based on user context (literacy, risk tolerance, life stage, emotional state, trust level, interaction history). Two users never see the same "home screen."

2. **Conversational + Visual Integration** — No "chat tab" and "dashboard tab." The conversation IS the interface. Charts materialize inline within the conversation. Navigation is emergent, not menu-driven.

3. **Hyperpersonalization as Architecture** — Not a feature layer on a generic app. The entire UI stack reasons about the user. Every component accepts user context as input. Individual-level, contextual, real-time personalization — not segment-based.

### Multi-Modal Voice + Screen Hybrid

The app uses a voice-first, screen-synced interaction model with a 4-stage entity pipeline:

- **Voice Agent** (Gemini 3.1 Flash Live) — Schema-agnostic NLU. Extracts **decomposed** entities from speech: base number and scale word are kept separate (LLM outputs `number: "80"`, `unit: "thousand"` — never does the multiplication itself). Also classifies coarse intent, detects emotional signals, and for what-if utterances extracts `modifier` (more/less/without/instead/percent) and `reference_field` (which existing entity is being operated on). ~1,730 token prompt, stable across journeys.

- **Entity Normalizer** (deterministic, Layer 1.5) — Canonicalizes the Voice Agent's decomposed output into algo-ready values. Uses `indian-word2number` + custom Hinglish lookup table (~50 Hindi number words, ~15 unit words) for number parsing, deterministic multiplication (number × unit multiplier), period resolution via contextual defaults (income→monthly, CTC→annual, EMI→monthly), and currency defaulting (INR, preserving explicit USD/EUR). Output: `{value, currency, period}` contract. **<2ms, zero GPU, 100% deterministic, no hallucination risk.** For the ~5% of entities where rules can't resolve confidently, falls back to the Orchestrator (which already has full context) rather than deploying a separate SLM.

- **Entity Mapper** (backend) — Maps normalized entities to product schema (D1-D8 + catchall). Deterministic rules for simple cases (age→D6 bracket, city→D5 tier), lightweight LLM for ambiguous cases. **Critical routing split:** `data_input`/`correction` intents → persist canonical values to DB; `what_if` intents → route as transient params to Brain (DB unchanged).

- **Brain / Orchestrator** (Gemini 3.1 Flash, text-only) — The reasoning layer. Selects appropriate math tools, handles what-if scenarios, manages contextual adjustments. **Never computes itself** — delegates all math to a deterministic, tested tool library.

- **Math Tool Library** — Preset, versioned, unit-testable functions (ef_target, allocation_split, compound_interest, etc.). Guarantees mathematical correctness by keeping LLMs out of the computation path.

### Latency Management: Bridge-Then-Inject

Since voice model pauses during tool execution, the system returns a mock acknowledgment instantly (~50ms) and uses bridge audio while the Orchestrator processes in the background:
- **T3 (standard, ~75%):** Mock response → bridge audio fills 3-5s while Orchestrator works (2-5s). Seamless.
- **T4 (deep reasoning, ~25%):** Bridge + "still working" pings for 5-10s processing. For complex what-if, multi-variable recomputation.

### Entity Pipeline

```
Voice Agent (Layer 1) → Entity Normalizer (Layer 1.5) → Entity Mapper (Layer 2) → Brain (Layer 3)
   Decomposed entities     Number × unit multiply       Schema mapping (D1-D8)     Reasoning + tool selection
   Coarse intent           Period/currency defaults      Persist vs transient       What-if recomputation
   Emotional signal        Hinglish normalization        Catchall bucket            Never computes
   modifier + ref_field    <2ms deterministic            Intent-based routing       Orchestrator dispatch
```

The Voice Agent is deliberately thin and schema-agnostic. It extracts raw decomposed entities — number and scale word separated, never multiplied — with a `raw_span` fallback per entity for downstream recovery. All business logic, schema mapping, and phase-aware routing lives in the backend. This keeps the voice prompt stable across journeys (J01, J02, J03+).

## First Journey: Emergency Fund (J01)

5-phase flow: Discovery → Assessment (8-dimension user profiling) → Target Setting → Fund Architecture (3 liquidity layers) → Contribution Planning. Users move non-linearly across phases. The Brain manages phase transitions and recomputation as user context evolves.

---

## Key Design Constraints

- **Brain never computes** — mathematical correctness guaranteed by deterministic tools
- **Unit contract throughout pipeline** — every monetary value carries `{value, currency, period}`, no naked numbers
- **Value-before-ask** — every data request preceded by demonstrated value from prior data
- **Adapt framing, never change the math** — AI adapts explanation depth, vocabulary, tone; never alters financial conclusions
- **What-if as transient computation** — "what if" scenarios don't persist to DB; data_input does
