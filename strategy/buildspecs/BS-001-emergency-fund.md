## 1. Product Context

> This section provides the receiving agent with the minimum viable context about CarrotFin needed to build this flow correctly. The receiving agent has zero prior CarrotFin knowledge.

### 1.1 What CarrotFin Is

CarrotFin is an AI-native personal finance advisor for Indian consumers. The AI is not a feature inside the app — it *runs* the experience. Every screen, every recommendation, every interaction is generated by the AI based on who the user is, what they need right now, and what they should do next. The interface is composed at render time from a palette of adaptive components and structural rules — there is no canonical "home screen" to design. Conversation and visualization are one integrated paradigm: charts materialize inside the conversation, recommendation cards ARE the AI's advice rendered visually, and users never navigate between a "chat tab" and a "dashboard tab."


[Source: company-context.md]

### 1.2 Governing Design Axioms

Three structural constraints govern every screen, component, and interaction in CarrotFin. All three apply to the Emergency Fund flow.

**Axiom 1 — Adaptive Composition** *(Highest Priority)*

Every touchpoint asks: "What should this user see right now, and how should it be assembled?" The interface is composed at render time by the AI from a palette of adaptive components and structural composition rules. The AI selects, orders, and configures components based on user context — financial literacy, risk tolerance, life stage, emotional state, trust level, and interaction history.

*EF example:* A 25-year-old first-time earner on the Emergency Fund assessment sees two cards with generous spacing, plain-English labels, and encouragement copy. A 40-year-old portfolio optimizer sees four compact cards with precise financial terminology, auto-expanded attribution detail, and delta metrics.


**Axiom 2 — Conversational + Visual Integration**

The AI speaks through the interface itself. There is no "chat tab" and "dashboard tab." A chart appears because the AI decided this user needs to see their spending trend right now. Navigation is emergent, not menu-driven.

*EF example:* When the AI reveals the emergency fund target, it speaks ("Your target is ₹4.2 lakh") while the Target Card with attribution strip materializes simultaneously in the stream — the spoken word and the visual element are one integrated moment, not sequential.


**Axiom 3 — Hyperpersonalization Is the Architecture**

Personalization is not a feature layer on top of a generic app. The entire UI stack is built to reason about the user. Every component accepts user context as input — not just "data" but "who is looking at this data." There are no personas. There is a continuous, multidimensional user context space.

*EF example:* The Attribution Strip in Phase 3 renders as a tiered linear arrow diagram for users with literacy ≥3, but degrades to a plain-text conversational breakdown for users with low literacy signals — same data, fundamentally different rendering based on who's looking.

**Conflict resolution hierarchy:** When axioms conflict, resolve in priority order: Adaptive Composition > Conversational+Visual Integration > Hyperpersonalization.

### 1.3 Interaction Paradigm

#### Surface Types Used in This Flow

| Surface Type | Definition | EF Flow Usage |
|:---|:---|:---|
| **Conversational Stream** | The primary interaction surface — a vertical flow where the AI speaks and the user responds. Messages can contain visual elements inline (charts, cards, forms). User responses can be structured (taps, sliders, chip selections), not just text. | Primary surface for Phases 1–5. All dimension capture, all reveals, all confirmations happen within the stream. |
| **Composed Surface** | A full-screen interface assembled by the AI from components, without a conversational thread. AI-assembled, not pre-designed. | Used for the Home Surface EF Goal Card after flow completion — a persistent, non-conversational card showing fund status. |
| **Ambient Layer** | Background intelligence surfaced proactively — nudges, alerts, opportunities. Delivered via push notifications, in-app banners, or conversation-starter cards. | Post-flow: salary-day reminders, milestone celebrations, momentum nudges, recalibration signals. Not the primary interaction mode during the setup flow. |

#### Modality Rules for This Flow

| Journey Phase | Leading Modality | Why |
|:---|:---|:---|
| Phase 1: Discovery | Conversational stream (voice or text). Open-ended, exploratory. | Discovery needs dialogue — the AI asks clarifying questions, builds understanding progressively. |
| Phase 2: Assessment | Stream + inline structured inputs. Multi-info capture per turn. | Data-dense phase. User provides information; AI captures via range pickers, sliders, chips — not open text fields. |
| Phase 3: Target Setting | Conversation → visual pivot. AI narrates while screen renders the Target Card. Low user input. | The reveal moment. AI speaks; visuals carry the detail. User reviews, doesn't generate. |
| Phase 4: Fund Architecture | Conversation + inline visual. AI explains allocation while the Allocation Card materializes. | Instructional — AI is teaching. Visual + verbal simultaneous delivery. |
| Phase 5: Contribution Planning | Conversational setup → persistent action card. | AI proposes; user confirms. Ends with a deliverable (the Action Card). |

#### Voice + Screen Hybrid Modality

This flow operates in a **voice + mobile screen** hybrid modality — or screen-only (text+tap) if voice is disabled. Voice is a first-class I/O channel layered on top of the screen, not a separate mode.

**Core rules:**

1. **Screen-first, voice-layered.** Every interaction is designed for text+tap as the baseline. Voice affordances are additive. No interaction requires voice.
2. **Screen is the source of truth.** Voice is ephemeral; the screen persists. After any voice input, the captured value appears on screen. Confirmation happens via screen, not voice alone.
3. **Simultaneous, not sequential.** When the AI narrates, the screen renders in parallel — not after. The AI's spoken message is a headline summary; the screen always contains equal or richer information.
4. **Voice does not gate.** Every question that can be answered by voice can also be answered by tapping or typing.
5. **Privacy-aware activation.** Voice is user-activated (opt-in). The app does not default to voice-on.

**Voice input handling by data type:**

| Input Type | Voice UX | Screen UX | Confirmation |
|:---|:---|:---|:---|
| Categorical (employment type, city, age bracket) | User speaks: "I'm freelance" / "Bangalore" | AI maps to category, highlights the matched option chip | **Light Confirm (§C1):** Option highlights, AI echoes verbally, auto-proceeds unless user corrects |
| Numeric (income, obligations) | User speaks: "About 80 thousand" | Number appears as a confirm chip — prominent, editable | **Active Confirm (§C2):** Chip remains until user taps ✓ or edits. AI pauses: "80 thousand a month — that right?" Flow waits. |
| Open-ended narrative (D9 contextual factors) | User speaks naturally | Transcribed text in stream, AI responds conversationally | No explicit confirm — AI responds with its interpretation, user can correct |
| Sensitive data (health conditions) | AI offers text-input fallback: "You can type this if you'd rather not say it aloud" | Same capture, with explicit text-input affordance | Per input type. AI does NOT verbally echo sensitive data — screen-only confirmation |

**Voice-Off Mode:** All voice input falls back to tap+type; all AI narration falls back to on-screen text. No information is lost. Voice is an I/O layer, not a branching path.

#### Confirmation Patterns

Four patterns used across all phases, referenced by ID:

| Pattern | When | Mechanism | Friction Level |
|:---|:---|:---|:---|
| **§C1: Light Confirm** | Categorical voice input (employment, city, age) | AI maps to category, highlights option, echoes verbally. Auto-proceeds. | Lowest — no tap required |
| **§C2: Active Confirm** | Numeric voice/text input (income, obligations) | Confirm chip with ✓ and ✏️. AI echoes with context. Flow pauses until user acts. | Medium — requires explicit user action |
| **§C3: Summary Confirm** | Phase-gate transition (e.g., Assessment → Target Setting) | Summary card materializes with all captured data as scannable rows. Two CTAs: "Looks right" / "Let me adjust." Each row tappable for edit. Flow waits. | Highest — deliberate friction at consequential moments |
| **§C4: Edit-in-Place** | User wants to change a previously given answer | Row transforms into editable field matching original input method. Update triggers live downstream recalculation. AI acknowledges the change. | Medium — triggered on demand, not imposed |

### 1.4 Design System Semantics

> Design parameters that define the experience feel. These are product intent — analogous to corner radius or font size.

#### Colour Semantics

The colour system extends Material Design 3 with semantic tokens that carry meaning beyond decoration:

| Token | Semantic Use |
|:---|:---|
| `warmPositive` (warm amber) | Positive attribution (e.g., "+2 months" in the attribution strip), the Instant Access liquidity layer, upward range changes, confirmed states |
| `coolNegative` (blue-slate) | Negative attribution (e.g., "−1 month"), the Stable Reserve liquidity layer, downward range changes |
| `neutralMid` (muted sand) | The Quick Access liquidity layer — between warm and cool. Cognitive middle ground. |
| `conditionalSurface` (warm cream) | D3 Prompt Card background — visually distinct from standard card surfaces to signal optionality |
| `conditionalBorder` | D3 Prompt Card dashed border — reinforces the "this is optional" visual signal |

**Dark mode note:** M3 handles most roles via `ColorScheme.fromSeed(brightness: Brightness.dark)`. The directional colour tokens (warm/cool) lighten in dark mode to maintain contrast.

#### Motion Philosophy

Motion uses **spring-based physics** — not linear easing. Springs give the UI a physical, breathing feel that responds to conversational pacing.

**Spatial motion (spring parameters):**

| Animation | Stiffness | Damping | Overshoot | Notes |
|:---|:---|:---|:---|:---|
| Card materialisation (slide-up + fade) | 300 | 0.8 | Subtle | All phase-gate cards, stream cards |
| Pill fill (BYP dimension capture) | 400 | 0.75 | Snappy | Smaller element = snappier spring |
| Number morph (₹ amounts, ranges) | 250 | 0.85 | Smooth | Counter animation with spring easing |
| Staggered entry (attribution rows, milestone rows) | 300 | 0.8 | Subtle | 60ms delay between items |

**Effects motion (non-spatial):**

| Animation | Curve | Duration | Notes |
|:---|:---|:---|:---|
| Colour transitions (warm/cool pulse) | easeInOut | 300ms | BYP directional colour changes |
| Opacity fade (card dismiss, skip) | easeOut | 200ms | Fade out on dismiss |
| Content crossfade (edit transitions) | easeInOut | 250ms | In-place content swap |

**Guiding principle:** "The UI breathes with the conversation." Animations respond to voice pacing — not robotic or canned.

#### Number Formatting

- ₹ symbol: prefix, no space → `₹1,20,000` (not `₹ 1,20,000`)
- Indian number system → `₹1,20,000` (not `₹120,000`)
- Monthly amounts → `₹6,000/mo`
- Ranges → `₹50K–80K` (en-dash, abbreviated)

### 1.5 Composition Grammar — Global Rules

> These rules govern how the AI assembles the interface. Phase-specific composition rules are embedded in §3.

#### Agent Architecture

The Adaptive Engine coordinates four agent concerns. This defines the structural contract between them:

| Agent | Receives | Produces | Downstream |
|:---|:---|:---|:---|
| **Conversational** | User utterance + dialogue history + current user state | Response text + extracted data signals (0–N dimensions per turn) | State agent (batch update); Composition agent (conversational context) |
| **Computation** | Captured dimension values (all 8 + D9) | ₹ target, allocation splits, milestone dates, contribution amounts | Composition agent. Semi-deterministic — formulaic core with LLM-calibrated range resolution (see note below). |
| **Composition** | User state + journey state + computed values + conversational context | Component selection + configuration: which component, which state, which adaptive variant | Rendering layer. Bounded by guardrails below. |
| **State** | Dimension updates from conversational agent + session signals | Current user-state snapshot: 4 adaptive dimensions with confidence bands + all captured journey data | All other agents (read-only). Persistent across turns. |

**Communication:** Agents exchange structured JSON context blocks — append-friendly, not replace. The composition agent receives full context and makes holistic rendering decisions. No hardcoded enum handoffs.

**Computation agent semi-deterministic note:** The 8-dimension sizing model (§3.2.2) specifies impact as ranges (e.g., "Govt/PSU: 3–4 months"). The computation agent uses a formulaic core for all deterministic outputs (zone widths, milestone dates, allocation splits, contribution projections), but the base EF target calibration within dimension ranges requires LLM-assisted contextual reasoning — e.g., where within the 3–4 month govt/PSU range a specific user falls depends on sub-dimension signals (employer stability, tenure, sector volatility). This is "semi-deterministic": the formula structure is fixed, but the input calibration has a reasoning step. *The precise calibration logic and weighting formula are out of scope for this BuildSpec and will be specified in a dedicated computation model artifact.*

**Batch capture:** A single user utterance can update 0–N dimensions simultaneously. *Example:* "I'm 34, salaried, two kids in Bangalore" → the conversational agent extracts age, employment type, dependents, city in one parse. The state agent receives all four as a batch. The BYP indicator animates multiple pill fills in a single staggered sequence (60ms offset per pill).

#### Component Mandate Vocabulary

| Term | Meaning |
|:---|:---|
| **Mandatory** | Component type is available and should be offered to the composition agent for this phase. Agent decides exact timing, sequencing, and configuration. |
| **Conditional** | Surfaces only when the specified trigger condition is met. Agent decides if/when the trigger fires, except where marked DETERMINISTIC. |
| **DETERMINISTIC** | A hard business rule. Agent has no discretion — the rule fires automatically when conditions are met. |
| **Optional** | Agent may choose to surface based on conversational context. No trigger rule. |

#### Viewport and Density Rules

| Literacy Band | Max Cards per Viewport | Card Content Padding | Section Gap | Card Vertical Margin |
|:---|:---|:---|:---|:---|
| 1–2 (low) | 2 | 20dp | 16dp | 12dp |
| 3 (mid) | 3 | 16dp | 12dp | 8dp |
| 4–5 (high) | 4 | 16dp | 8dp | 8dp |

**Emotional override:** If emotional state is *overwhelmed*, reduce max cards by 1 regardless of literacy.

**Pinned components excluded:** The BYP Progress Indicator does not count toward the viewport card limit — it occupies its own fixed zone.

#### Conflict Resolution Hierarchy

When rendering rules conflict, resolve in this order:

> **Adaptive Composition > Conversational+Visual Integration > Hyperpersonalization**

Specific precedence rules:

1. **Mandatory components always render** — but their configuration (expanded/collapsed, detail level) is adjusted by user state.
2. **Conditional triggers are hard rules** — DETERMINISTIC triggers cannot be suppressed by user state.
3. **Optional components yield first** — when density limits are exceeded, optional components are dropped before conditional ones are collapsed.
4. **Animation pacing adjusts independently of layout density** — spacing and spring parameters are separate dimensions.

#### User-State Inference

The composition agent's rendering decisions depend on four adaptive dimensions, each inferred by the state agent via LLM contextual reasoning — not hardcoded scoring formulas:

| Dimension | Signal Type | Output | Update Cadence |
|:---|:---|:---|:---|
| **Financial literacy (1–5)** | Vocabulary precision, question depth, response speed, engagement with detail | Band: 1–2 / 3 / 4–5 + confidence | Per conversational cluster |
| **Trust level** | Data voluntarily shared, recommendations accepted, return frequency | Level: New → Warming → Trusting → Dependent + confidence | Per session boundary |
| **Emotional state** | Word choice, pacing, topic sensitivity, explicit statements | State: Calm / Anxious / Overwhelmed / Excited | Per turn (most volatile) |
| **Engagement mode** | Input modality distribution, dwell time on visuals, CTA response latency | Mode: Conversational / Visual / Hybrid | Per cluster (last 3–5 turns) |

Low-confidence dimensions default to the more conservative rendering option.

## 2. Flow Overview

### 2.1 What This Flow Is

The Emergency Fund Setup flow is CarrotFin's first-run experience and primary V1 user journey. It takes a new user from zero financial context to a fully personalized emergency fund plan: a target amount with transparent attribution, a 3-layer liquidity allocation strategy, and a monthly contribution plan with milestones.

Users arrive with no emergency fund or an unsized one; they leave with a personalized target, allocation strategy, and contribution plan. The EF assessment doubles as user profiling — the 8 assessment dimensions seed the user profile for all future features, eliminating a standalone onboarding form.

**The flow is advisory-only.** CarrotFin recommends instrument *types* (savings account, liquid fund, short-term FD) but never names specific banks, funds, or products. Execution happens outside the app. This is a deliberate product constraint — not a temporary limitation.

[Source: J01-emergency-fund-setup.md, DD01-ef-as-onboarding.md, DD02-advisory-only-model.md]

### 2.2 User Segments

| Segment | Key Characteristics | Behavioral Signals Relevant to This Flow |
|:---|:---|:---|
| **S1: Young Professionals (25–30)** | First-time earners/investors, single, salaried, metro, high AI comfort. ₹5L–25L annual income. | Likely no existing EF. High engagement with progressive disclosure. Need confidence, not information — they know they should save but want the AI to tell them they're making the right choice. |
| **S2: Young Families (28–35)** | Recently married or new parents, dual income, need protection + planning. | EF urgency spikes at life transitions (marriage, first child). Social obligation buffer (D3) highly relevant — family support expectations affect fund sizing. May have partial savings but unsized. |

[Source: user-insights.md]

#### First-Run Entry Point

The app does NOT land the user directly into the EF flow. Instead:

1. The app presents a **goal discovery surface** — "What would you like to work on first?" — with visible options:
   - 🛡️ **Build my safety net** (Emergency Fund) ← Active in V1
   - 🎯 **Plan for a goal** ← Placeholder (visible, not interactive)
   - 📈 **Grow my wealth** ← Placeholder
   - 🧾 **Understand my finances** ← Placeholder
2. Only the EF path is live. Other paths show "Coming soon" when tapped.
3. This shell means onboarding is architecturally goal-agnostic — future goal types slot in without restructuring.

[Source: J01-emergency-fund-setup.md §1B, DD01-ef-as-onboarding.md]

#### Re-Entry Matrix

The flow is non-linear by design. Users arrive at different stages of readiness. The AI detects the user's stage and routes them to the appropriate entry point — never forcing redundant steps.

**Detection mechanisms:**
- **App state (returning users):** Serialized per-beat journey state from prior sessions — the AI knows exactly which phases/beats are complete and which data points are captured.
- **Conversational probing (new users with prior knowledge):** The AI's opening question and follow-ups surface signals about existing knowledge, allowing dynamic routing.

**Five entry scenarios:**

| Scenario | Signal | Entry Point | AI Framing |
|:---|:---|:---|:---|
| **A: Fresh User** | No prior sessions. No EF signals. | Phase 1 (Discovery). Full flow. | Discovery conversation — establish relevance and urgency. |
| **B: Has Existing Savings** | "I have some savings" / "I've been saving but not sure if it's enough" | Skip Phase 1. Enter Phase 2 in "health check" framing. | "Let me check if your savings are sized right for your situation." Respects existing effort — frames as validation, not "start from scratch." |
| **C: Has Known Target** | "I calculated I need about ₹5L" / "I read 6 months of expenses" | AI offers validation. If accepted → Phase 2, comparing AI's number with theirs at Target Setting. If declined → skip to Phase 4 with user's number. | "That's a solid starting point. Online calculators often miss a few factors — mind if I run a quick check?" User autonomy first — never force redundant assessment. |
| **D: Returning Mid-Flow** | Serialized journey state from prior session | Resume from last completed beat. Quick summary card of captured data before continuing. | "Welcome back. Last time, we figured out your income profile. Want to pick up where we left off, or start fresh?" State persists per-beat, not per-phase. Staleness rule: if >30 days, re-verify key data. |
| **E: Has Existing EF** | "I have ₹3L in savings and a ₹2L FD. Is that enough?" | Skip Phase 1 + early Assessment. Pre-fill existing fund data. Run assessment → show gap or surplus. | "You have ₹5L total — solid foundation. Let me check if it's the right amount." Emphasises validation and optimization. Reinforces proactive behavior. |

**Key architectural principles for re-entry:**

1. **Each phase is data-dependent, not sequence-dependent.** Phase 3 needs the 8-dimension data, not Phase 2's "completed" flag. If data is available from any source, the phase can execute.
2. **The AI is the router.** No hardcoded phase-sequence enforcement. The state machine supports arbitrary entry points.

[Source: DD05-multi-stage-re-entry.md, J01-emergency-fund-setup.md §0B]

**In V1:** Conversational onboarding (EF as first-run), 8-dimension adaptive assessment, personalized target with attribution, social obligation buffer, 3-layer allocation recommendation (instrument type only), contribution plan with milestones, insurance gap advisory flag, app shell with EF + placeholder goals.

**Deferred to V2:** Monitoring & nudging surface, crisis withdrawal + rebuild mode, recalibration on life event (active UX — trigger logic in V1 data model), insurance gap dedicated journey, transaction execution, archetype shortcuts.

## 3. Experience Specification

> This section defines the phase-by-phase user experience. Each phase specifies: the AI's job, the component palette available, adaptive rendering rules, behavioral guardrails, and the phase-exit condition. The AI is not scripted — phases define intent and constraints; the AI decides exactly what to say and when.

> **Prerequisite for all phases:** The user has selected "🛡️ Build my safety net" from the §1B goal selection surface (see §2.3). §1B is the app's entry point, not a journey phase.

---

### 3.1 Phase 1: Discovery

**User question:** "Do I actually need an emergency fund? I thought my savings/investments cover me."

**AI's job:** Establish relevance and urgency without fear-mongering. Earn the right to ask for data by delivering value first.

**Surface type:** Conversational stream (generative — no composed surfaces).

**Trust level at entry:** New (zero interactions, zero data).

[Source: J01-interaction-specs.md Phase 1, CG01-composition-grammar.md §Phase 1]

#### 3.1.1 Agent-Improvised Phase

Phase 1 has no pre-defined component specs. The AI composes Discovery screens from the design system's primitives using the guidance below. This is deliberate — Phase 1 is the most conversational, least structured phase and should be the most agent-driven. The AI improvises the rendering within the design system's tokens and the composition grammar's global rules (§1.5).

[Source: CG01-composition-grammar.md §1 Phase 1]

#### 3.1.2 AI Intent & Guidance

The AI receives the following context and constraints. It decides the exact words, order, and tone.

**Goal:** Make the user feel that an emergency fund is personally relevant and worth building. Phase 1 ends when the user expresses willingness to assess their needs (transition to Phase 2).

**Approach:**

1. **Open with a direct, conversational question** about whether they have an emergency fund today. Offer structured tappable options (e.g., "Yes, I have one" / "Sort of" / "Not really" / "Not sure what counts") while accepting free-text or voice input. Use §C1 Light Confirm (see §1.3) for the selection.

2. **Adapt based on the user's response** — this is where re-entry routing happens:
   - *Already has savings* → Acknowledge, reframe as a health check, route toward Phase 2 with validation framing ("let me check if it's sized right"). This is Re-Entry Scenario B (see §2.3).
   - *Already has a target* → Acknowledge, offer optional validation. If accepted → Phase 2. If declined → accept their number and skip to Phase 4. User autonomy first. This is Scenario C.
   - *Has an existing EF to review* → Skip Discovery entirely. Route to Phase 2 with "gap check" framing. This is Scenario E.
   - *No fund / uncertain* → Proceed with Discovery steps below.

3. **For users who need Discovery:** establish what qualifies as an emergency. Use an interactive element (e.g., quick-sort card with ✅ Emergency / ❌ Not Emergency scenarios). Examples: job loss ✅, planned vacation ❌, medical emergency ✅, new phone ❌, urgent home repair ✅, festival shopping ❌. This is value delivery at zero data cost — the user learns something before the AI asks for anything.

4. **Myth-busting is conditional, not default.** Only surface a myth rebuttal when the user signals a specific misconception:
   - "I'll use my credit card" → Surface credit card APR cost (36–42% APR; ₹2L emergency → ₹3L+ within a year)
   - "My investments cover me" → Surface SIP interruption and forced-sale risk (selling equity in a crash crystallizes losses; stopping SIP destroys compounding)
   - "My family will help" → Gentle reframe: informal safety nets are declining with urbanization; family support should be a bonus, not a plan
   Use loss framing per behavioral principle §1.1 (loss aversion). Tone: matter-of-fact, not preachy.

5. **Bridge to Assessment:** Close with an invitation ("let me figure out your actual number — not a generic rule") and a single primary CTA. The phrasing is an invitation, not a commitment gate.

**Literacy calibration:** Phase 1 is the AI's first signal window for inferring the user's financial literacy level (see §1.5 User-State Inference). The AI observes:
- Response speed (fast taps → high comfort; long pauses → lower comfort)
- Input modality (structured options vs. free-text vs. voice)
- Vocabulary in free-text ("SIP" vs. "that monthly saving thing")

These signals carry forward and shape vocabulary, explanation depth, and component density in all subsequent phases.


[Source: J01-interaction-specs.md Phase 1 LLM Guidance Prompt, behavioral-framework.md §1.1, §4.4, emergency-fund-philosophy.md §7]

#### 3.1.3 Component Palette

Phase 1 is agent-improvised, but the following component types are available for the AI to compose from:

| Component | Purpose | Behaviour |
|:---|:---|:---|
| **Structured option chips** | Opening question response options | 3–4 tappable options below AI message. §C1 Light Confirm. Voice input maps to nearest option. |

### 3.2 Phase 2: Assessment

**User question:** "Okay, I need one. How much do I actually need?"

**AI's job:** Capture the 8 sizing dimensions through adaptive conversation — not a form. Deliver value after each cluster of captures. The user should feel their picture being built, not interrogated.

**Surface type:** Hybrid — pinned "Building Your Picture" progress indicator (top of viewport) + conversational stream below.

**Trust level at entry:** New → Warming (trust advances during this phase as the user shares data and receives value-after insights).

[Source: J01-interaction-specs.md Phase 2, DD06-assessment-stream-primary.md]

#### 3.2.1 Stream-Primary Architecture

The Assessment is conducted entirely within the conversational stream. There is no composed surface, no form-like layout, no modality switch.

The conversational stream IS the product differentiation; a composed surface collapses the value-after trust cadence into a single "fill this in" transaction. Inline components (option chips, confirm chips, range pickers, sliders) handle all structured input within the stream.

[Source: DD06-assessment-stream-primary.md]

#### 3.2.2 The 8 Sizing Dimensions

The EF target is derived from 8 adaptive dimensions — not a static formula. This is the core advisory logic.

| # | Dimension | What the AI Captures | Sizing Impact |
|:---|:---|:---|:---|
| **D1** | Income Stability | Employment type + employer stability / income pattern | Govt/PSU: 3–4 months base. Large private: 4–6. Startup: 6–8. Freelancer: 8–12. Gig: 6–10. |
| **D2** | Dependency Load | Who depends on the user's income — nature matters, not just count | Aging parents with no insurance ≠ working adult sibling. Each dependency type has different risk weight. |
| **D3** | Insurance Coverage Quality | Health insurance level; term insurance flag | Adequate health (₹10L+ cover) + term: reduce by 1–2 months. No health insurance: add ₹2–5L buffer. |
| **D4** | Fixed Obligations | Rent/EMI, school fees, premiums — as ₹ amount or % of income | High fixed-to-variable ratio = larger fund required. |
| **D5** | City Tier / Cost of Living | Metro vs. Tier 2 vs. Tier 3 | Metro essential family spend: ₹60K–1.2L/mo. Tier 2: ₹25–50K. Tier 3: ₹15–35K. |
| **D6** | Age Bracket | Career stage, not exact year | 22–27: 3–4 months (high re-employability). 28–35: 6–8. 35–45: 8–12 (peak responsibility). |
| **D7** | Household Income Sources | Sole earner vs. dual/multiple | Dual-income salaried = lowest risk. Single income + variable = highest risk. |
| **D8** | Health Profile | Pre-existing conditions — SENSITIVE | Chronic conditions, family history of critical illness → increase buffer. AI does NOT echo verbally. |
| **D9** | Open / Contextual | User-volunteered factors outside D1–D8 | "I also support my brother's education" / "We have a legal case pending" → +0.5 to +2 months buffer. |

**Illustrative output of dimension interplay:**
> "Your target is ₹4.2L (7 months) because: you're a single earner (↑), have 2 dependents including an uninsured parent (↑), but have stable government employment (↓) and good personal health insurance (↓)."

[Source: emergency-fund-philosophy.md §2, J01-interaction-specs.md Phase 2 LLM Guidance Prompt]

#### 3.2.3 AI Intent & Capture Rules

**Goal:** Capture all 8 dimensions (+ optional D9) through natural conversation so the computation agent can calculate the user's personalized EF target. Phase 2 ends when all 8 dimensions are captured and confirmed at the §C3 summary gate.

**Core capture rules:**

1. **No fixed sequence.** Dimensions may be captured in ANY order the conversation flows. If the user says "I'm 34, salaried, two kids in Bangalore" in one turn, capture D6 + D1 + D2 + D5 simultaneously. Acknowledge naturally, don't re-ask what was already given.

2. **Batch capture.** A single user utterance can update 0–N dimensions simultaneously. The AI parses and captures all data points from a single statement. The BYP indicator animates multiple pill fills in a single staggered sequence (simultaneous fills with 60ms offset per pill).

3. **Group related dimensions conversationally.** D1+D7 together as "income profile"; D2+D3 together as "who depends on you and how protected they are." Grouping should flow naturally — never feel like a survey.

4. **Adaptive follow-ups.** Some dimensions need depth based on initial answers:
   - D1 "Salaried" → ask employer type (govt/PSU, large private, startup, mid-size)
   - D1 "Own business" → ask tenure (3+ years vs. newer/variable)
   - D2 "Aging parents" → ask about their health insurance status
   - D2 "Children" → clarify age bracket if not already stated
   The AI decides whether a follow-up is needed based on what's already been said. If the user already provided the info ("I'm salaried at TCS"), don't re-ask.

5. **D9 triggers asynchronously.** Whenever the user volunteers context outside D1–D8, acknowledge it, ask a brief clarifying follow-up if needed, and incorporate as a conservative buffer (+0.5 to +2 months). A 9th pill appears in the BYP indicator. Do NOT create a structured question for D9 — respond conversationally and make a judgment call.


[Source: J01-interaction-specs.md Phase 2 LLM Guidance Prompt, CG01-composition-grammar.md §0.3 Batch Update Protocol]

#### 3.2.4 Confirmation Patterns Used

Phase 2 uses three of the four confirmation patterns (see §1.3 for full definitions):

| Input Type | Pattern | Behaviour |
|:---|:---|:---|
| **Categorical** (employment type, city, age bracket, yes/no) | **§C1 Light Confirm** | AI maps to category, highlights matched option chip. AI echoes verbally. Auto-proceeds unless user corrects. No tap required. |
| **Numeric** (income, fixed obligations) | **§C2 Active Confirm** | Confirm chip renders with ✓ and ✏️ affordances. AI echoes with interpretive context ("80 thousand a month — that right?"). Flow PAUSES until user confirms. Uses range pickers/sliders as primary input, not blank text fields. |
| **Sensitive** (D8 health) | **Text fallback** | AI explicitly offers text input as alternative to voice: "You can type this if you'd rather not say it aloud." AI does NOT verbally echo health details — screen-only confirmation. "Prefer not to say" is always valid → use conservative estimate. |

**Income handling specifically:** Offer range brackets (₹30K–50K, ₹50K–80K, etc.) or accept a spoken/typed number. For variable income, help the user find a working estimate (midpoint of good/bad months). §C2 for the confirmed number.

**At phase gate (Assessment → Target Setting):** **§C3 Summary Confirm** fires after all 8 dimensions are captured. See §3.2.7.

#### 3.2.5 "Building Your Picture" Progress Indicator

Pinned horizontal strip between top navigation and conversational stream. Always visible during Phase 2. Read-only — not interactive.

**Purpose:** Tracks the 8 assessment dimensions (+ optional D9) as the AI captures them. Shows a running estimated target range that narrows in real time. This is the primary orientation mechanism — the user sees their picture being assembled, not a form being filled.

**Visual design:**
- 8 dimension pills (labelled: Income · Dependents · Protection · Obligations · City · Age · Health · Household), always rendered
- Running estimate text: "Your range: — months" → "Your range: X–Y months" → "Your target: N months (₹X.XL)"
- Does NOT count toward viewport card limit — it occupies its own fixed zone

**State progression:**

| State | Trigger | Visual Change | Motion |
|:---|:---|:---|:---|
| Start | Phase 2 entry | 8 empty pills. Estimate: "— months" | — |
| Dimension captured | AI captures any D1–D8 | Pill fills with animated pulse (○ → ✓) | Spring: stiffness 400, damping 0.75 |

### 3.3 Phase 3: Target Setting

**User question:** "What's my number? And why?"

**AI's job:** Reveal the personalised target with full dimension attribution. Make the number feel inevitable and personal — not arbitrary. This is CarrotFin's signature trust-building moment.

**Surface type:** Hybrid — conversational reveal + inline Target Card with attribution strip.

**Trust level at entry:** Warming (trust advanced during Phase 2 through value-after moments).

[Source: J01-interaction-specs.md Phase 3, DD07-attribution-strip-design.md]

#### 3.3.1 Modality Pivot

Phase 3 is a role reversal. The AI shifts from questioner (Phase 2) to narrator. The user shifts from data provider to evaluator. Three rules govern the modality:

1. **AI is narrator, not reader.** Narrate headlines and the editorial key insight. The screen carries data and attribution. The AI does NOT read the attribution strip aloud row by row.
2. **Silence is a tool.** After the reveal, go quiet. The user needs space to evaluate. Speak again only if the user asks a question or initiates an edit.
3. **Voice carries emotion, screen carries logic.** The spoken key insight is the emotional hook. The attribution strip is the logical proof. These render simultaneously, not sequentially (§1.3 voice rule: simultaneous, not sequential).

[Source: J01-interaction-specs.md Phase 3 Voice Handling Principles]

#### 3.3.2 AI Intent & Guidance

**Goal:** Reveal the user's personalised EF target with full attribution so they understand WHY the number is what it is. Phase 3 ends when the user accepts a target (or adjusts and re-confirms) and all post-target steps (D3 buffer, Starter Shield) are resolved.

**What the AI knows at this point:**
- All 8 dimensions (+ D9 if any) — confirmed at §C3 gate
- Computed target (₹ amount + months) — from the computation agent
- Attribution breakdown: each dimension's directional contribution (+ or − months)
- User's literacy level and emotional signals from Phases 1–2
- Re-entry context (Scenario B existing savings, Scenario C user's own target, Scenario E existing EF corpus)

**The Reveal Sequence:**

1. **Conversational setup:** Brief narrative pause before the Target Card materialises. Build a moment — don't jump straight to the visual. The pause distinguishes this from a calculator output.

2. **Target Card materialises inline** with: headline (₹ amount + months), attribution strip (DD07), and two CTAs ("Looks right ✓" / "Adjust an answer"). See §3.3.3 for full component spec.

3. **Divergence handling:** If the target diverges materially from the generic "3–6 months" advice, auto-expand the "Why is this different?" section within the card. If within 3–6 months, keep it tappable but don't auto-expand.

4. **Deliver ONE key insight** — the single most surprising or high-impact factor from the attribution. This is the AI's editorial voice: interpret the data, don't just read it. Choose the factor that would most change the user's mental model. *Example:* "The biggest thing people miss is the uninsured parent — that single factor adds a full month."

5. **Go silent.** Let the user evaluate. Do not prompt, rush, or filler-talk.

**Re-entry adaptations:**

| Scenario | Target Card Adaptation | AI Framing |
|:---|:---|:---|
| **B** (existing savings) | Headline includes gap analysis below attribution — "You have ₹X.XL — you're N% there." Gap shown AFTER attribution, not within it. | Key insight emphasises validation: "You've already built a solid foundation." |
| **C** (validation path) | Comparison row: user's target vs. CarrotFin's target. Modified CTAs: "Go with ₹X.XL" / "Stick with my ₹X.XL." | User autonomy first — the AI offers, never forces. |
| **E** (existing EF review) | Gap or surplus framing below attribution. | If surplus: "You're over-funded — the extra could work harder elsewhere." |

[Source: J01-interaction-specs.md Phase 3 LLM Guidance Prompt, DD05-multi-stage-re-entry.md]

#### 3.3.4 Attribution Strip (DD07)

The attribution strip is the "Show Your Work" mechanism — a vertical stacked diagram where each row shows one dimension's contribution to the target. It earns trust for a number the user will anchor their entire financial safety net to.

**Row anatomy:**

`[Factor label]   [Description]   [+/− N.N months]`

- 5–9 rows (D1–D8 + optional D9)
- D9 row appended at bottom, labelled with the AI-assigned context
- Visual treatment of D9 is identical to D1–D8

**Row tap behaviour (progressive disclosure — §1.5 Verification Anchors):**

| Tap | Action | Motion |
|:---|:---|:---|
| First tap | Level 2: one-sentence plain-English explanation of this factor's impact | Crossfade 250ms `easeInOut` |
| Second tap | Level 3: formula + data source. Third tap collapses. | Crossfade 250ms `easeInOut` |

**Collapsed state (default for literacy ≥3):**

Default collapsed shows **top 3 factor rows** + affordance: "Show all N factors." Tap expands to full row list.

**"Why is this different?" affordance:**

| Condition | Behaviour |
|:---|:---|
| Target diverges from 3–6 months generic range | Auto-expands |
| Target within 3–6 months | Tappable only — does not auto-expand |

> The composition agent decides the editorial "key insight" — the single most surprising/high-impact factor to narrate. The spec does not prescribe which factor is the key insight.

**Literacy fallback (DD07 — critical for low-literacy users):**

For users with literacy signals at band 1–2, the visual attribution strip is **replaced** by a plain-English conversational breakdown. The AI narrates the attribution directly in the conversational stream:

> *Example:* "Your number is 7 months — about ₹4.9L. Here's why: most people in a salaried job at a big company start at 5 months. Having a baby adds about a month. Your mother-in-law isn't covered by health insurance, so I added another month for that. Your household has two incomes, which brings it down a bit. All together: 7 months."

No visual strip renders. A persistent "Show me the breakdown" affordance lets the user recover the visual strip if desired. This fallback ensures the "Show Your Work" trust mechanism reaches ALL users, not just visually literate ones.

> **Engineering note (DD07):** Literacy band is inferred from a composite signal — not a user-declared field. Detection heuristic: user answered all assessment questions via tap (never typed), spent >median time on each step, and/or AI vocabulary calibration from Phase 1 indicates low financial literacy. The fallback is triggered when this composite signal is low, which maps to bands 1–2. The heuristic is defined inline — no external reference required.

### 3.4 Phase 4: Fund Architecture

**User question:** "Okay, I know the target. But where do I actually put this money?"

**AI's job:** Recommend how to allocate across 3 liquidity layers and explain the structure. No product execution — that belongs to Phase 5. Phase 4 ends when the user has mentally accepted the structure.

**Surface type:** Hybrid — conversational setup → inline Allocation Card with horizontal liquidity gradient strip.

**Trust level at entry:** Warming → Trusting (trust advances as the user accepts the allocation recommendation).

**Advisory ceiling:** Instrument TYPE only. No specific fund names, AMC names, or bank products. This is a hard constraint from DD02 (see §2.1).

[Source: J01-interaction-specs.md Phase 4, DD08-allocation-visual-design.md, DD02-advisory-only-model.md]

#### 3.4.1 AI Intent & Guidance

**Goal:** Help the user understand WHERE to put their emergency fund money across 3 liquidity layers. Phase 4 ends when the user confirms they understand and accept the structure. Do NOT discuss contribution amounts or timelines — that is Phase 5.

**What the AI knows at this point:**
- User's EF target (₹ amount + months) — confirmed in Phase 3
- Income pattern (variable vs. salaried, sole vs. dual earner) — from Phase 2
- Target size — determines DICGC trigger
- D3 social obligation buffer (if added) — treat as a separate sub-fund

**The "3 Speeds of Money" framing:**

The AI frames the allocation as three speeds, not three buckets. This primes the user's mental model before the visual appears:
- **Speed 1 (Instant):** 1–2 months in savings account or sweep-in FD. Available via ATM or UPI 24/7.
- **Speed 2 (Quick):** 2–4 months in liquid mutual fund. T+1 in general; up to ₹50K is near-instant for most funds.
- **Speed 3 (Stable):** Remaining months in short-term FD or debt fund. 1–3 business days. Highest idle yield.

**Adaptive layer weighting (income-pattern-dependent):**

| Income Pattern | Layer 1 (Instant) | Layer 2 (Quick) | Layer 3 (Stable) | Rationale |
|:---|:---|:---|:---|:---|
| Variable income / sole earner / gig | **Higher proportion** — more instant-access | Standard | Lower | Income is lumpy — need more buffer without any friction |
| Stable salaried / dual income | Standard | **Higher proportion** | **Higher proportion** | Regular income is a natural hedge — less needs to sit in lowest-yield layer |

**Explanation approach:**
- Lead with "why" not "what." The 3-layer structure exists because emergencies have different speeds.
- Adapt vocabulary to literacy level: high-literacy → "liquid fund is T+1 with near-instant for ₹50K"; low-literacy → "it's like an FD that you can get back the next business day."

**What the AI does NOT do:**
- Name specific banks, AMCs, or fund products
- Execute or suggest executing anything — that's Phase 5
- Present this as complex — the structure is simple once the "3 speeds" frame lands

[Source: J01-interaction-specs.md Phase 4 LLM Guidance Prompt, emergency-fund-philosophy.md §5]

**Zone label visibility:**

| Zone width | Label rendering |
|:---|:---|

| 56dp–79dp | Compact: ₹ amount only |
| < 56dp | Minimum width floor triggered (see below) |

**Minimum width rule:** When proportional sizing compresses any zone below 56dp (unreadable on mobile), apply a floor of 56dp and redistribute overflow across the other zones. | Zone | Fill | Semantic |
|:---|:---|:---|

| Quick Access | `neutralMid` (muted sand) | Neutral = middle ground, next-day |

### 3.5 Phase 5: Contribution Planning

**User question:** "How do I actually build this over time?"

**AI's job:** Translate the target into a monthly contribution plan. Frame it as a "self-EMI." Connect to the Starter Shield milestone from Phase 3. Hand the user a persistent Action Card (DD09) and an optional salary-day reminder. This is the journey's highest-stakes CTA — the user must leave the app and act.

**Surface type:** Hybrid — conversational setup → Contribution Plan Card → Action Card + reminder opt-in.

**Trust level at entry:** Trusting (trust advanced through Phases 2–4 via progressive value delivery and recommendation acceptance).

**Advisory ceiling:** Instrument TYPE only. No specific fund names, AMC names, or bank products. The user executes externally. This is a hard constraint from DD02 (see §2.1).

[Source: J01-interaction-specs.md Phase 5, DD09-contribution-plan-deliverable.md, DD02-advisory-only-model.md]

#### 3.5.1 AI Intent & Guidance

**Goal:** Turn the user's EF target into a concrete monthly action they will actually start. The plan is advisory only — CarrotFin is not executing transfers. Phase 5 ends when the user confirms their plan and receives their Action Card.

**What the AI knows at this point:**
- EF target (₹ amount + months) — confirmed at Phase 3
- Allocation structure (3 layers + amounts per layer) — confirmed at Phase 4
- Monthly income (range or working estimate) — from Phase 2 Assessment
- Income pattern (salary date predictability: fixed payday vs. variable) — inferred from D1 employment type
- Starter Shield milestone amount (1 month of essential expenses) — from Phase 3 §3-SS if triggered
- D3 social obligation buffer (if active) — has its own separate contribution line

**What the AI helps the user figure out:**

1. **How much per month, realistically, can they set aside?**
   - Ask for an amount. Don't assume — this is personal.
   - Once they give a number, compute: at that rate, time to full target.
   - Offer a comparison: "At ₹X more per month, you'd get there Y months sooner."
   - Recommend the faster path — show the cost of the gap in plain terms (the more months before they're funded, the more months they're exposed if something goes wrong).
   - For variable income users: reframe. Don't ask for a fixed monthly amount. Instead: "What's the minimum you'd commit on a slow month? On a good month, you can top up." Frame a floor — "target ₹X as your floor — whatever extra comes in, send it too."

2. **Frame it as a self-EMI.**
   The moment to introduce this frame is when the contribution amount is agreed. "Think of this like an EMI you pay yourself. The difference: this one builds a cushion instead of clearing a debt." The self-EMI frame leverages existing Indian mental models — EMI = automatic, non-negotiable, happens before spending.
   For variable income users, the frame flexes: "Set aside what you can on salary credit, aim for ₹X as your floor."

3. **Connect to the Starter Shield.**
   If the Starter Shield was shown in Phase 3 (§3.3.6), the first milestone in Phase 5 is the same number. Do NOT re-introduce it as new — reference it: "Your first milestone is the Starter Shield — ₹[X] — same one we talked about earlier." Show how many months at their chosen contribution rate until they hit the Starter Shield. This is the anti-paralysis anchor — it makes the full target feel reachable by shrinking the first step.

4. **Milestone timeline.**
   Show the full milestone path with estimated dates:
   - Starter Shield (1 month): at ₹[contribution]/mo, you hit this in [N] months — around [Month Year]
   - 3 Months Secure: [date]
   - Half-Year Shield: [date]
   - Fully Funded: [date]
   Dates are more motivating than "N months" — they create a concrete mental anchor.

**What the AI does NOT do:**
- Name specific banks, AMCs, fund products, or platform names for execution
- Execute or suggest executing any transfer — that is beyond the advisory ceiling
- Force a specific amount — the user chooses, the AI advises

[Source: J01-interaction-specs.md Phase 5 LLM Guidance Prompt, DD02-advisory-only-model.md, DD09-contribution-plan-deliverable.md, behavioral-framework.md §1.2]

> "Your plan is set. I'll keep an eye on things and reach out when something meaningful happens — a milestone hit, a life change, or a chance to optimise. You won't hear from me just to check in."

**This copy is verbatim-locked.** It must never be paraphrased, improvised, or truncated by the LLM. The agent has zero discretion on this phrasing.

**Design intent:**
- Tone: proactive, non-intrusive, purposeful — the CarrotFin monitoring-state voice
- "You won't hear from me just to check in" is the differentiation claim — distinguishes CarrotFin from notification-spam apps
- Delivered as a standard AI message bubble in the conversational stream (Body Large, 16sp/400)
- Motion: fade-in, opacity 0→1, 200ms `easeOut`

**After the exit message:** Monitoring state begins. The EF goal card on the home surface updates to show: fund target, current amount (₹0 for new users, or existing corpus for Scenario B/E), next milestone, contribution plan summary, Action Card entry point, and progress bar (current amount ÷ target).

[Source: CP05-contribution-plan.md §Exit Confirmation Message, J01-emergency-fund-setup.md §1A]

#### 3.5.11 Phase-Exit Condition

Phase 5 ends when ALL of the following are completed:

1. **Contribution plan confirmed:** User taps "Confirm my plan →" (or voice equivalent).
2. **Action Card delivered:** Card materialises in stream and is saved to EF goal card on home surface.
3. **Reminder resolved:** User either accepted the salary-day reminder, declined it (notification permission denied or dismissed), or skipped it.
4. **Exit message rendered:** Verbatim §1A copy (§3.5.7) delivered.

After the exit message, the V1 conversational flow is complete. Monitoring state begins — ongoing nudges, milestone celebrations, and recalibration signals are delivered via the ambient layer (push notifications, home surface EF card) as defined in §1A nudge philosophy. Full monitoring UX is V2 scope.

## 7. Unresolved Tensions & Assumptions

> This section documents design tensions and assumption dependencies that affect the J01 Emergency Fund flow. Tensions are genuine conflicts between opposing forces — not bugs or missing decisions. Assumptions are untested or partially validated beliefs that the design depends on. Both are flagged for the dev agent so it doesn't unknowingly resolve a tension prematurely or build on an invalidated assumption.

### 7.1 Design Tensions Affecting J01

#### T1: Simplicity vs. Financial Rigor

| Force A | Force B |
|:---|:---|
| Users want 1-tap answers and jargon-free language. A 25-year-old first-time investor doesn't want to learn about expense ratios. | Financial accuracy requires nuance, caveats, and context. Oversimplifying without explaining fees, lock-in, and tax implications is irresponsible. |

**J01 impact:** The 8-dimension Assessment (§3.2) and the Attribution Strip (§3.3.4) are the primary surfaces where this tension manifests. The adaptive literacy bands (1–2 / 3 / 4–5) are the current mitigation — but the thresholds between bands are untested. The literacy fallback (replacing the attribution strip with plain-English narration for bands 1–2) is the strongest resolution mechanism. If user testing shows that literacy 3 users also find the visual strip confusing, the fallback should be extended.

**Resolution requires:** User testing with at least 3 literacy levels, measuring comprehension AND conversion-to-action at each complexity tier.

---

#### T2: Conversational vs. Visual Primacy

| Force A | Force B |
|:---|:---|
| Some decisions need dialogue — exploration, education, emotional processing. | Some insights need charts — spending trends, goal projections. Describing a spending breakdown in words is objectively worse than a chart. |

**J01 impact:** The modality pivot in Phase 3 (§3.3.1) is the clearest test case — the AI shifts from questioner to narrator, and the Target Card carries the data while voice carries the emotion. Phase 4's simultaneous "3 speeds" narration + gradient strip (§3.4.2) is another. Both depend on the user following the AI's modality lead. If users resist the pivot (e.g., try to ask questions during the reveal silence), the Phase 3 interaction design may need adjustment.

**Resolution requires:** Prototype the modality handoff patterns and observe whether users follow the AI's lead or fight to stay in one modality.

---

#### T3: Personalization vs. Privacy

| Force A | Force B |
|:---|:---|
| Deep personalization requires deep data access. The more the AI knows, the better the advice. | Indian users are wary of sharing financial data with any app, let alone an AI. |

**J01 impact:** The trust ramp in Phase 2 (§3.2.10 BV2) is the primary mitigation. The value-after cadence (§3.2.6) earns the right to ask progressively sensitive questions. D8 (health) is the most sensitive dimension — the text-input fallback and "prefer not to say" option (§3.2.4) are protective mechanisms. The key unknown is the trust ramp speed: how many value-delivering interactions before users share income? Health data? If the ramp is slower than designed, Phase 2 abandonment will spike at D4 (fixed obligations) or D8 (health).

**Resolution requires:** Longitudinal study of data-sharing behavior across at least 50 users over 30+ days.

---

#### T6: Behavioral Nudging vs. User Autonomy

| Force A | Force B |
|:---|:---|
| Behavioral science tools (loss framing, anchoring, friction design) genuinely help users make better financial decisions. A well-timed nudge prevents an impulsive redemption. | Every nudge is a subtle manipulation. The AI has asymmetric power — it controls what the user sees and how it's framed. Where does "helpful guidance" end and "dark pattern" begin? |

**J01 impact:** Phase 1 myth-busting (§3.1.2) uses loss framing, which is a nudge. Phase 3 Starter Shield (§3.3.6) anchors the user toward an achievable first step — another nudge. Phase 5 self-EMI framing (§3.5.1) leverages the existing EMI mental model — also a nudge. The guardrails (§3.1.6: "genuine financial reality, not manufactured urgency") constrain the intensity. The ethical test: would the user, fully informed about the nudge mechanism, still want the AI to use it?

**Resolution requires:** User testing of nudge patterns at different intensity levels. Observe whether the user feels helped or pressured, and whether the nudge leads to financial outcomes the user later endorses.

---

#### T7: Verification Depth vs. Cognitive Load

| Force A | Force B |
|:---|:---|
| "Show Your Work" builds trust — users who can verify the AI's math trust its judgment faster. Transparency earns the right to advise. | Most users don't want to see formulas. Too much math creates anxiety or overwhelm. The point of an AI advisor is that users don't have to do the math themselves. |

**J01 impact:** The Attribution Strip's 3-level progressive disclosure (§3.3.4) is the primary resolution attempt — headline → reasoning → formula → data sources. The "fire extinguisher effect" (the option to drill down is sufficient for trust even if never used) is the design hypothesis. The literacy fallback (plain-English narration for bands 1–2) is the safety valve. The key unknown: how many users actually drill down to Level 3, and whether the presence of Level 3 is sufficient for trust without use.

**Resolution requires:** Prototype "Show Your Work" at 3 disclosure depths. Measure trust scores, tap-through rates, and whether the presence of a "see calculation" affordance improves trust even when not used.

---

#### T8: Conversational Assessment Pacing vs. User Efficiency

| Force A | Force B |
|:---|:---|
| The conversational assessment format is the product differentiator. Each question + value-after moment builds trust and earns permission for the next question. Rushing collapses the trust ramp. | Some users — especially returning users, literate users, or re-entry Scenario B/C/E — may find 10–12 conversational turns slow. They want the number. A "quick mode" would be significantly faster. |

**J01 impact:** The V1 design (DD06, embedded in §3.2.1) commits to stream-primary for all users. This is the right default — first-time users need the trust ramp. But as repeat usage grows and Scenarios B/C/E become dominant, the tension will intensify. The re-entry matrix (§2.3) already skips Discovery for these cases, but the Assessment itself remains conversational. If completion time > 5 minutes AND beat 2 abandonment exceeds 30%, a "Quick Assessment" variant (single structured form with the same data types) should be prototyped.

**Resolution requires:** Track Assessment completion time and abandonment rate by beat. If thresholds are exceeded, design and A/B test a "Quick Assessment" variant.

### 7.2 Assumption Dependencies

> These are untested or partially validated assumptions that the J01 design depends on. If an assumption is invalidated during development or testing, the corresponding design elements should be re-evaluated.

| ID | Assumption | Status | J01 Impact | Risk if Wrong |
|:---|:---|:---|:---|:---|
| C4 | An adaptive/personalized UI meaningfully outperforms a static dashboard for financial planning | ⬜ Untested | The entire composition grammar (§1.5), all adaptive behaviour tables (§3.x.7/§3.x.8), and the user-state inference model depend on this being true. | If false, the entire product architecture needs rethinking. The adaptive engine becomes unnecessary overhead. |
| C5 | Conversational AI integrated with visuals is better than chat + dashboard as separate surfaces | ⬜ Untested | The stream-primary assessment (§3.2.1), inline card materialisation throughout, and the voice+screen hybrid modality (§1.3) all depend on this. | If false, separation of chat and dashboard may improve usability. | C6 | Users trust AI financial advice enough to act on it (beyond information consumption) | ⬜ Untested | Critical for Phase 5 conversion — does the user actually set up the auto-transfers after receiving the Action Card (§3.5.5)? The "confidence gap" (users consume advice but don't act) would require embedded execution or human advisor integration. |

| EF-15 | Progressive milestone framing (Starter Shield → 3 Months → Full) drives better completion rates than showing the full target upfront | ⬜ Untested | Core to Starter Shield (§3.3.6), milestone timeline (§3.5.3), and the anti-paralysis design strategy. | If false, Starter Shield is unnecessary complexity. Simpler "here's your target, here's your plan" may perform equally or better. |

## 8. Glossary

> Every CarrotFin-specific term used in this BuildSpec. The receiving agent has zero prior CarrotFin context — all domain-specific vocabulary must be defined here.

| Term | Definition |
|:---|:---|
| **Action Card** | Post-plan-confirmation deliverable card (DD09) with numbered step-by-step instructions for external fund setup. Persists to the EF goal card on the home surface. See §3.5.5. |
| **Adaptive Composition** | The highest-priority design axiom (§1.2). The UI is composed at render time by the AI from a palette of adaptive components based on user context. There is no canonical screen layout. |
| **Advisory ceiling** | Hard constraint (DD02): CarrotFin recommends instrument types only — never specific bank names, AMCs, fund products, or platform names. See §4.1 M1. |
| **Allocation Card** | Phase 4 phase-gate card (§3.4.2). Visualises the recommended 3-layer liquidity allocation using a horizontal gradient strip. |
| **Ambient layer** | Background intelligence surface — push notifications, in-app banners, conversation-starter cards. Used post-flow for salary-day reminders and milestone celebrations. See §1.3. |
| **Attribution Strip** | Vertical stacked diagram within the Target Card (DD07). Each row shows one dimension's directional contribution (+ or − months) to the EF target. 3-level progressive disclosure. See §3.3.4. |
| **BYP (Building Your Picture) Progress Indicator** | Pinned horizontal strip during Phase 2. Tracks 8 dimension pills and a running estimated target range. See §3.2.5. |
| **Composition agent** | The agent that receives user state, journey state, computed values, and conversational context, and produces component selection + configuration decisions. See §1.5. |
| **Composed surface** | A full-screen interface assembled by the AI from components, without a conversational thread. Not used during the J01 assessment. See §1.3. |
| **Computation agent** | The agent that receives captured dimension values and produces semi-deterministic outputs: ₹ target, allocation splits, milestone dates, contribution amounts. Formulaic core with LLM-calibrated range resolution for base target (see §1.5 semi-deterministic note). |
| **Confirm Chip (§C2)** | Active confirmation pattern for numeric inputs. Renders ✓ and ✏️ affordances. Flow pauses until user confirms. See §1.3. |
| **`conditionalBorder`** | Dashed border token used on the D3 Prompt Card to visually signal optionality. |
| **`conditionalSurface`** | Warm cream fill token used on the D3 Prompt Card to distinguish optional cards from standard surfaces. |
| **Conversational agent** | The agent that processes user utterances, generates response text, and extracts data signals. See §1.5. |
| **Conversational stream** | The primary interaction surface — a vertical flow where the AI speaks and the user responds. Visual elements render inline. See §1.3. |
| **`coolNegative`** | Blue-slate colour token. Semantic use: negative attribution, Stable Reserve liquidity layer, downward range changes. See §1.4. |
| **D3 buffer** | Optional social obligation buffer — a separate fund for weddings, ceremonies, and family support. D3 Prompt Card and D3 Confirmed Card. See §3.3.5. |
| **D9** | Open/contextual dimension — user-volunteered factors outside the 8 core dimensions. See §3.2.2. |
| **DETERMINISTIC** | A hard business rule with no agent discretion. Fires automatically when conditions are met. See §1.5 Component Mandate Vocabulary. |
| **DICGC footnote** | Deposit Insurance and Credit Guarantee Corporation disclosure. Rendered as a factual note below the Allocation Card when EF target > ₹5L. Not a warning. See §3.4.5. |
| **Edit-in-Place (§C4)** | Confirmation pattern for editing previously given answers. Inline editor matching original input method. Triggers downstream recalculation. See §1.3. |
| **EF Goal Card** | Static, persistent card on the home surface showing fund target, current amount, next milestone, and contribution plan summary. See §3.5.12. |
| **Horizontal Liquidity Gradient Strip** | Single continuous horizontal bar (DD08) within the Allocation Card. Three colour-coded zones proportional to ₹ amounts, representing the liquidity spectrum. See §3.4.3. |
| **Light Confirm (§C1)** | Lowest-friction confirmation pattern for categorical voice input. AI echoes verbally and auto-proceeds. See §1.3. |
| **Literacy fallback (DD07)** | For users with literacy signals at band 1–2, the visual attribution strip is replaced by a plain-English conversational breakdown. See §3.3.4. |
| **Monitoring state** | Post-V1-flow state. Ongoing nudges, milestone celebrations, and recalibration signals via the ambient layer. Full UX is V2 scope. See §3.5.7. |
| **`neutralMid`** | Muted sand colour token. Semantic use: Quick Access liquidity layer, cognitive middle ground. See §1.4. |
| **Phase gate** | Flow-blocking transition point. The AI does not advance until the user explicitly confirms via CTA. See §3.2.7, §3.3.10, §3.4.6, §3.5.11. |
| **Self-EMI** | Contribution framing that leverages the Indian EMI mental model — "an EMI you pay yourself." See §3.5.1. |
| **Starter Shield** | Conditional milestone framing (§3.3.6). Reframes a large target as an achievable first step (1 month of essential expenses). DETERMINISTIC trigger. Visual continuity between Phase 3 and Phase 5. |
| **State agent** | The agent that receives dimension updates and session signals and produces the current user-state snapshot (4 adaptive dimensions with confidence bands + all captured journey data). See §1.5. |
| **Summary Confirm (§C3)** | Highest-friction confirmation pattern. Summary card with all captured data as scannable rows. Used at phase-gate transitions. See §1.3, §3.2.7. |
| **Target Card** | Phase 3 phase-gate card (§3.3.3). Headline ₹ target + months + attribution strip. Multi-state: default, recalculated, Scenario B/C/E, Starter Shield. |
| **Trust ramp** | Progressive trust model: New → Warming → Trusting → Dependent. Trust level advances through demonstrated value, accepted recommendations, and data sharing. See §1.5 User-State Inference. |
| **Value-after** | Post-data-capture insight moment. The AI tells the user what their data means for their estimate, earning the right to ask more questions. Three value-after moments in Phase 2. See §3.2.6. |
| **`warmPositive`** | Warm amber colour token. Semantic use: positive attribution, Instant Access liquidity layer, upward range changes, confirmed states, Starter Shield glow. See §1.4. |
